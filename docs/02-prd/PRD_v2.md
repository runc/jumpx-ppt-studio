# PRD v2 · Jumpx Slides — 可掌控配方的 AI 幻灯片生成器

> 2026-05-31。v2 在 课件2 完整PRD v1（JumpX AI 实战营 Week04 课件） 的**生成内核之上**叠加两件事:**① 把"配方(skill)"变成用户可看/可改/可换的一等资产;② 落成单机、无账号的独立产品。**
> 配套文档:[`UX_DESIGN.md`](../01-design/UX_DESIGN.md)(交互)、[`SKILL_CONTROLLER.md`](../04-agent-control/SKILL_CONTROLLER.md)(配方控制器)、[`TECH_SPIKES.md`](../03-implementation/TECH_SPIKES.md)(技术验证)、[`IMPLEMENTATION_PLAN.md`](../03-implementation/IMPLEMENTATION_PLAN.md)(实施)。

---

## 0. 一句话定义

> Jumpx Slides 是一个**单机的 AI 幻灯片生成器**:输入主题/材料,由一个自主 Agent(deepagents on LangGraph)**驱动一份"配方(skill)"**生成可见成品幻灯片;过程在关键岔路口征询用户(选模板、出图/HTML);并且**把"内容怎么生成"的配方交给用户——可查看、可调整、可上传别人的配方,本地生效**。给的是结果,且**结果背后的逻辑你能掌控**。

### 0.1 与 v1 的关系(重要)
- **v1 的生成内核全部成立、不推翻**:agent 驱动 skill、两个交互点、门禁、Context Lock、双输出、结果优先。
- v2 的增量 = **配方即产品**(skill 可见/可改/可换)+ **单机无账号**。
- 生成主流程**固定不变**(除非发现流程缺失);用户**只能改"内容层",改不动"流程/契约/机制"**(见 §12)。

---

## 1. 背景与问题

市面 AI PPT 工具是**黑盒**:你不知道它为什么这么生成,也无法掌控生成逻辑;想要不同结果只能反复抽卡。我们已有生产级 skill(`ai-slide-producer`),且 deepagents 原生支持 skill。**v2 的差异化:把生成逻辑(配方)透明化、可编辑、可替换——想要不同的 deck,最根本的杠杆是换/调配方,而不是抽卡。**

---

## 2. 定位与目标用户

- **定位**:单机、无账号的独立桌面/本地 Web 应用。一个"把控制权交给用户"的 PPT 生成器。
- **核心用户**:想要**可控、可定制生成逻辑**的内容创作者 / 讲师 / 顾问 / 进阶用户;以及想沉淀、复用、分享自己"配方"的人。
- **非目标用户**:要复杂财务表格、要多人实时协作、要严格品牌终稿的人。

---

## 3. 核心需求

1. 把粗糙输入变成**可见、可展示、可继续改的幻灯片成品**(v1 内核)。
2. **看懂"内容为什么这样生成"**——因为是配方指令在控制。
3. **掌控生成逻辑**:想要不同版本(更厚/更薄、换叙事、换领域脑子),通过**调配方**拿到,而非抽卡。
4. **换配方 = 换一位帮你写 slides 的"专家人格"**(领域背景 × 叙事风格 × 厚薄)。
5. 能**上传**别人的配方(如从 GitHub 下载的)、本地生效;能**下载/导出**自己的配方。

---

## 4. 产品架构

```
用户（网页 UI · 画布为主、AI 为副驾）
   ↕
自主 Agent（deepagents on LangGraph）——规划(todo)、岔路口中断征询、流式
   ↓ 驱动（动态加载当前 active 配方）
配方 / Recipe（= 一个 skill = 一个 zip）
   ├─ 可改层（内容怎么来）：叙事 / 写作风格 / 素材吸收 / 背景知识 / 厚薄
   └─ 锁定层（契约+机制+安全）：schema / 门禁 / 渲染 / 系统安全规则
   ↓ 产出
可见成品（HTML slides / 图片 slides）+ 可继续编辑/重生
```

**三层模型(开放程度不同):**
| 层 | 内容 | 开放 |
|---|---|---|
| ① 素材 / Context | 本项目输入 + agent 压成的 context_pack | 用户数据,完全开放 |
| ② 配方 / Skill | 叙事 / 写作风格 / 素材吸收 / 背景知识 / 厚薄 | **可看 + 受约束可改 + 可上传** |
| ③ 契约 / Mechanics | slide_plan/style_lock schema、九步门禁、layout 枚举、渲染、安全 | **锁定** |

**关键认知:**
- **配方 = skill = zip**:zip 就是分发协议(我们的 `ai-slide-producer.zip` 已是现成格式)。
- **动态加载**:agent 在**每次生成开始时 fresh 实例化**、读当时的 active 配方 → "新配方只管下一份、不碰在跑的"。
- **单机本地**:无账号、无多租户;状态本地 SQLite 持久化;配方库是本地目录;分享走带外(GitHub/文件)。

---

## 5. 核心概念 / 实体

- **配方 (Recipe)** = 一个 skill 目录(SKILL.md + references + schemas + assets + scripts),打包为 `.zip`。带 **manifest**(名称/版本/作者/目标 contract 版本/可编辑范围声明)。
- **可改层** = 配方里允许用户改的部分(§12 明确边界)。
- **锁定层** = 契约/机制/安全;用户改不动(改了也被忽略,见 §12 吸收规则)。
- **配方库** = 本地一堆配方;有"当前使用(active)"标记;支持 内置 / 我的 / 已导入。
- **配方控制器** = 菜单入口的查看/编辑/重载界面(简洁旋钮 + 进阶 Markdown)。
- **验证 skill** = 一个内置的 meta-skill,判定候选配方是否"符合契约 + 安全",不过则不允许使用。

---

## 6. 产品范围

### 6.1 MVP(本期目标)
- **生成全流程**:输入 → 大纲 → 选模板 → 出图/HTML → 渲染(逐页流式)→ 完成 → 导出。两个 ask 交互点 + 大纲确认。
- **配方控制器**:查看配方;**简洁旋钮**编辑(叙事/语气/素材吸收/背景知识/厚薄)+ **进阶 Markdown** 编辑(可改 references,锁定层只读);保存即"对下一份生效"。
- **配方库(本地)**:内置数套人格化配方;**上传(.zip)/ 编辑 / 下载(.zip)/ fork(本地复制改)/ 选用**。
- **配方校验门**:编辑/上传 → 验证 skill 把关(契约符合 + 安全),不过不可用。
- **持久化**:本地 SQLite,生成可中断、可恢复。

### 6.2 非 MVP / 后续(明确不在本期)
- 真正的"用户提交共享库 / 市场 / 评分 / 审核 / 身份"——**需要后端 + 账号,与单机无账号冲突,后续阶段再议**。MVP 的"分享"= 导出 zip,用户自己用 GitHub 等带外传播。
- 多人协作、品牌终稿、复杂表格。

---

## 7. 核心原则

1. **结果优先**:最终给可见成品,不是建议。
2. **先确认再放大**:Brief→大纲→风格,关键节点过门禁。
3. **Context Lock 防漂移**:确认即冻结、反复读取。
4. **结构与驱动分离**:skill 管"怎么做",deepagents 管"自主做 + 交互"。
5. **画布为主、AI 为副驾**:刻意避开通用聊天框外观;岔路口决策占主舞台、不挤在对话里。
6. 【新】**透明可控**:生成逻辑对用户可见、可调。
7. 【新】**契约不可破**:用户能改"内容怎么来",改不动"产物结构/门禁/渲染/安全";违反者不允许使用。
8. 【新】**新生成用新配方**:改配方只影响之后开始的 Slides,不打断在跑的。

## 8. 主流程

```
[配方控制器]（菜单常驻：查看/调/换/上传/下载 当前及其他配方）
       │（active 配方 在生成开始时被读取并固定到本次 run）
       ▼
1 输入主题/材料（+ 篇幅/受众/语气前置 chip）
2 Agent 列 todo + 规划                         ← 可见进度/流式
3 生成大纲 → ⛔ 用户确认（左树/右预览，可改）
4 ❓选模板（缩略图网格，套到你封面，推荐预选）
5 ❓出图 / HTML（同页两版并排，诚实取舍）
6 生成 slide plan → 内容审核
7 渲染（骨架屏 → 逐页点亮 + 活动流，可中断）
8 质量检查
9 完成 → 导出（HTML / PDF / 图片 / PPTX）
10 继续迭代（改这一页 / 局部重生）
```

## 9. 交互规格

| 交互点 | 时机 | 形态 | 默认 |
|---|---|---|---|
| 大纲确认 | 大纲生成后 | 左树/右故事板预览,可改 | 等待确认 |
| 选模板 | 大纲确认后 | 7 套缩略图网格(套到你封面),推荐预选 | teaching-clean |
| 出图/HTML | 选模板后 | 同页两版并排 + 取舍标签 | HTML |
| **配方控制器** | 菜单常驻 | 查看/旋钮改/进阶Markdown改/上传/下载/换 | 当前 active 配方 |
| 改这一页 vs 改配方 | 渲染后 | 改这份=单页编辑/Mark up;改配方=控制器 | 区分清楚 |
| 昂贵操作授权(出图) | 调付费出图前 | 需用户点继续 | 需确认 |

> 实现:ask 交互点 = deepagents `interrupt_on`(respond);决策面渲染在**主舞台覆盖层**,不弹聊天气泡。

## 10. UI / UX(详见 UX_DESIGN.md)

- 画布为主 + AI 副驾(三态待办 + 对话 + 可折叠活动流);底部胶片缩略图轨。
- 5 屏:输入起步 / 大纲编辑(故事板)/ 选模板(套封面预览)/ 选输出(同页两版)/ 工作台(渲染高光 + 完成 + 导出)。
- 渲染高光:骨架屏 → 逐页点亮 + 活动流自动展开、完成后收起(产品高光,区别于竞品黑盒等待)。
- 配方:独立**叙事页**("配方=会写某类 deck 的脑子" + 人格画廊)+ 编辑器(旋钮 + 进阶 Markdown)。
- 设计系统:纸感暖白 + 墨绿强调(外壳),内容用 teaching-clean(主-客对比);明/暗可切。

## 11. 门禁(Gate)

生成 5 门(v1):Brief 确认 → 大纲确认 → 内容审核 → 风格锁定 → 结果可见。
【新】**配方校验门**:配方被编辑/上传 → 验证 skill(契约符合性 + 安全审查 + 干跑产物 schema 校验)→ 不过则**不允许用于生产**(保留原配方,给可读报错,提供"恢复默认")。

## 12. 配方系统详规

### 12.1 可改边界(已拍板)
**可改(②层):** ① 叙事结构 ② 写作语气/风格 ③ 素材吸收方式 ④ 背景知识(配方自带、可复用的领域脑子) ⑤ 厚薄。
对应文件:`references/03-strategist.md`(叙事)、`05-writer.md`(语气/厚薄)、`02-context-pack.md`(吸收)、`references/background.md`(背景知识,新增)+ density 参数。
**锁定(③层):** `SKILL.md` 管线/门禁、`schemas/*`、`assets/templates`、`scripts/*` 渲染机制、系统安全规则。

### 12.2 上传 = 只吸收可改层
用户上传一个(完整)配方 zip 时:**只抽取可改层并到我们当前的锁定基座上**;**不执行/不采纳用户版的锁定层**。若用户改了锁定文件 → 忽略,但**回告用户"对 X 的改动被忽略(锁定)"**。
> 副作用(好的):用户的可改层**永远骑在我们最新的锁定基座**上 → **不存在陈旧 fork**;同时**堵死"替换门禁/系统提示"的注入路径**。

### 12.3 下载 = 完整版
下载给完整 skill(锁定 + 可改),便于离线/GitHub 查看与编辑;**重新上传时只吸收可改层**。下载产物应**标注 editable vs locked**(供离线编辑者知道什么会生效)。

### 12.4 验证(meta-skill 硬门)
分层:结构 lint(manifest/可改层完整、可达)→ 安全审查(明显注入)→ 廉价干跑(生成 1–2 页)→ `validate_slide_plan.py` 校验合 schema/layout 枚举。任一不过 → 不可用。

### 12.5 schema 升级时
我们升级 layout/结构后,**验证 skill 把库里所有配方(的已吸收可改层)重跑一遍**,不通过的标记**失效**(失效配方不可选用,提示用户更新)。配方 manifest 的 contract 版本用于判定与提示。

### 12.6 安全
上传/导入的配方 = **不可信内容**。硬保障**不是**验证 skill(LLM 注入检测可被绕过),而是:
- **收紧 agent 工具面**:只暴露我们定义的工具(build_slides_html / generate_image / choose_*),**不给任意 shell/网络**;
- **文件访问沙箱进 workspace**;
- **系统提示的契约+安全规则压在配方之上**。

---

## 13. 成功标准

- 任意 5–20 页可稳定生成;HTML 可翻页、风格一致;最终给可见结果。
- **换配方能产出可感知差异的 deck**(不同人格/厚薄/叙事)。
- 用户能**看懂并调整**生成逻辑;编辑配方→下一份生效。
- 上传/下载/fork 配方可用;校验门能挡住坏配方。
- 生成可中断、可恢复(单机重启不丢)。

---

## 14. 非目标

复杂表格/财务模型;多人实时协作;品牌手册终稿;**MVP 不做用户提交共享市场/账号/审核/评分**;不开放契约层编辑。

---

## 15. 主要风险

| 风险 | 缓解 |
|---|---|
| **默认生成质量是命门**(验证只保合法、不保好) | 先把默认配方做到惊艳;可改范围窄而直观,差异可预测 |
| deepagents token ~20×,且"改配方→验证→生成"循环烧钱 | 单机由用户自带 key 承担;提供廉价预览(只重生大纲/单页);昂贵出图前授权 |
| **导入配方安全/注入**(单机本地有文件/工具权限) | 工具面收紧 + workspace 沙箱 + 系统提示压顶(§12.6);验证 skill 仅作合规辅助 |
| 验证非确定性(LLM 干跑有方差) | 结构 lint 为主、干跑为辅;失效判定可复验;给"恢复默认"逃生口 |
| 配方理解门槛(改指令→预判输出难) | 简洁旋钮覆盖 80%;裸 Markdown 作 power-user 逃生口;廉价预览补反馈 |
