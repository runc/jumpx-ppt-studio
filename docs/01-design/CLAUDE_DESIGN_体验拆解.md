# Claude Design 制作原型的交互体验拆解

> 2026-05-31。我们用 Claude Design 实际做了一个 AI 幻灯片 WebApp 的界面外壳(从零上下文 → 设计系统 → 主工作台 + 输入页 → handoff 给 Claude Code 落地)。本文记录并拆解它**在"做原型"这件事上的交互体验**——既是使用复盘,也是一次同域产品(Claude Design 本身就是"对话生成设计/deck"的 AI 产品)的交互学习。
> 图片引用的是制作过程中的实际截图(图1–图7),保存在与本文同次会话的记录里。

---

## 0. 一句话总评

Claude Design 把"做设计"**重新定义成了一个 agent 任务**:你不是在画布上拖控件,而是像跟一个会设计的同事对话——它先**澄清方向**、**列计划**、**边做边给你看**、**自我校验**、**给多方案让你挑**,最后**打包交付给写代码的 agent**。它把"agent 工作流"那套(计划/流式/人在回路/验证/交接)完整搬到了设计场景,且产物质量(版式、配色、字体、克制度)明显高于"通用 AI 外观"。对我们做 AI 幻灯片产品而言,它几乎是一面镜子:我们要给用户的"生产交互",它正用在"生产 UI"上。

---

## 1. 心智模型:对话驱动,不是画布驱动

传统设计工具(Figma/Canva)是**画布驱动**——你面对一张空画布,工具栏里找控件。Claude Design 是**对话驱动**:

- 左栏是**对话 + agent 工作流**(你说需求、它回计划与产物);
- 右栏是 **Design Files** 画布,但你**几乎不直接在上面拖拽**,而是"看结果 + 局部圈选告诉它改哪"。
- 画布顶部空状态写着 "Creations will appear here / Start with a sketch",底部一句 TIP:"The best design system is the one nobody notices."——基调就是"你说,我做"。

> **对我们的启示**:这正是我们产品要的姿态——"画布为主、AI 为副驾"。区别是它的画布放设计稿,我们的画布放幻灯片;但"对话驱动 + 画布呈现 + 局部圈选改"这套骨架可以直接借。

---

## 2. 开场不急着做,先"方向对齐"(图1)

粘进需求后,它**没有马上画**,而是弹出一个 `questions_v2` 澄清面板("动手前的方向对齐(主要是视觉基调)"),把会"决定成败"的几个点一次问清:整体气质、明暗、色温、强调色、字体、副驾语气、信息密度、要几个方向、骨架屏感觉、产品名。

交互设计上有几处很讲究:
- **结构化多选,不让你打字**。每题给 3–4 个**带描述的选项**(如"墨绿/松绿(沉稳、专业、不像 AI 套皮)"),而不是开放문本框——降低输入熵、所见即所得。
- **每题都有 `Decide for me` 和 `Other…`**:既给"我懒得想,你定"的智能默认,又留自由表达的口子。部分选项还有 `Explore a few options`(让它出多方案)。
- **默认值预高亮**(如"副驾语气"默认选了"像专业助手")——智能默认覆盖多数决策。
- **色温这种纯视觉的题,用色块缩略图**(暖中性/冷中性/纯灰三块)而非文字,再配一题文字版确认。
- 面板顶部先**复述它的理解**("先说我的理解,确认我没跑偏"),把澄清和确认合一。

> **这是 review-before-build 的教科书实现**,也是我们 UX_DESIGN 里"富交互卡片 gate + 智能默认"的活样板。**直接可抄到我们的"输入起步页"**(篇幅/受众/语气用 chip 选,不打字)。

---

## 3. 边做边可见:计划 + 流式 + 自我校验(图2)

确认方向后,左栏像一个**透明的 agent 工作台**逐步展开:

- `update_todos` → 一个**可折叠的 todo 清单**(Updated todos / Writing ×2 / Done, Verify),让你看到它"在第几步"。
- `write_file` → 写出设计文件;`done` → 标记完成。
- **最有意思的是 `fork_verifier_agent`**:它**派生了一个验证 agent 自检产物**,真的查出了一个 CSS 类名冲突——`.todo`(容器)和 `.task todo`(待办状态)撞类——并**自动改名 `.todo-list` 修好**(图2 的 "Good catch… Renaming the container to .todo-list")。
- 生成中有 `Scrambling…` / `Thinking…` / `Generating questions…` 等**有区分的加载态**,不是一个干巴巴的转圈。

> **启示**:① todo 三态可视化(我们工作台已照做);② "派生验证 agent 自检"这套,和我们后端"reviewer/style-guard 自动门"是同构的,值得在前端也暴露给用户看(增强信任);③ 加载文案要**具体**("正在排图文双栏")而非 "Loading…"。

---

## 4. 不给一个答案,给多方向 + 差异说明 + 精准追问(图3、图6)

它**默认出 2 个方向并排**(我们在澄清阶段选了"给 2 个方向对比"),而且做得很完整:

- 顶部一段**设计系统说明头**(三栏:设计系统 + 方向A + 方向B 各一段话,带色板色块);
- 每个方向有**标签**(`opt-tag` 方向A/方向B)+ 标题 + 一句描述(如"裱框主角 · 底部胶片轨 · 活动流更突出");
- 两版都做成**同一个"渲染中"状态**,把骨架屏、逐页点亮、三态待办、活动流"一次摆全"——**用最能体现产品高光的状态来对比**,而不是随便摆个静态页;
- 给完不丢给你拉倒,而是**抛 3 个精准追问**("选 A/B/混搭?缩略图竖排还是底部胶片?副驾优先级认可吗?"),把决策收敛到可一句话回答。

输入页那屏(图6)同理:方向 A 居中聚焦 / 方向 B 桌面预演,各有侧重、各配说明。

> **启示**:多方案不是"生成 3 张让你看",而是**"可比的同状态 + 差异说明 + 收敛式追问"**。这套"判断—收敛"的交互,正是我们产品在"选模板""选布局"等岔路口该有的。

---

## 5. 改:点画面局部直接说(Mark up,图4)

这是把"富展示"变成"真交互"的关键一招:

- 顶部常驻提示 "**Click and tell Claude what to change · drag to draw**";
- 你**点高保真图的任意局部**(或拖拽框选一块),弹出 `Mark up` 浮层:`Describe the issue or suggestion…` + 两个动作 **`Add comment`(只批注)/ `Send to Claude`(触发 AI 改)**;
- 作用域 = 你圈选的那块,**局部改、不整页重来**。

顶栏还有一排模式:**Mark up(M)/ Edit(E)/ Tweaks(T)/ Comments(C)/ Present / Share**,以及缩放 100%。即"看/批注/直接编辑/微调/评论/演示/分享"分得很清楚。

> **启示**:这正是我们"改这一页"触点的范式——**点幻灯片上的元素 → 说要改什么 → 局部重生**。而且它把 **"只批注" 和 "让 AI 改" 拆成两个按钮**(对应 HITL 的 respond vs 不执行),很克制。我们落地时要解决的硬点是:**幻灯片元素要能映射回 slide_plan 字段**,才能做到"点谁改谁、页间隔离"。

## 6. 交付:一个给"写代码 agent"的 handoff 包(图7 / 实测)

最能体现它"为生产闭环设计"的,是 handoff。导出给的不是一张图,而是一个 instruction + 一个 `tar.gz`:

- 它生成一句话让你粘进 Claude Code:**"Fetch this design file, read its readme, and implement… Implement: Jumpx Slides/主工作台.html"**;
- 抓下来是 `AI PPT-handoff.tar.gz`(路径 token 即授权,无需 API key),解开是:
  - `README.md` —— **专门写给 coding agent 的指令**:先读 chat 记录(意图在那)、找主文件读完、跟着 imports 读、**用适合目标仓库的技术像素级复刻**(别照抄原型内部结构)、有歧义先问、"别为了理解去截图"。
  - `project/CLAUDE.md` —— **设计系统记忆**(色板 hex、字体、布局基底决策),给代码侧持续用。
  - `chats/chat1.md` —— **完整对话记录**(意图与最终落点)。
  - `project/Jumpx Slides/*.html` —— HTML 原型本体。

> **启示**:这是"设计→代码"跨界的范本——**把意图(chat)、规范(design system)、产物(html)、给机器的说明(README)打包**,而不是丢一张图让人猜。我们产品的"交付"环节也该想:交给用户的不只是 deck,还应有"可继续编辑/可重生"的结构化包(我们的 export-contract 已有雏形)。

## 7. 做得好的(亮点清单)

1. **不急着画,先澄清** —— 把返工成本压在最前面。
2. **结构化选择 + 智能默认** —— 几乎不用打字就能定方向。
3. **过程透明** —— todo/流式/验证 agent 全看得见,信任感强。
4. **多方向 + 收敛追问** —— 不是"生成完拉倒",而是帮你做决定。
5. **局部圈选改** —— direct manipulation,改一处不动全局。
6. **产物质量高** —— 严格守住"暖白纸感 + 墨绿 + 中文字体"的约束,没滑进"紫色渐变 AI 套皮";甚至自查 CSS 冲突。
7. **为生产闭环交付** —— handoff 包 + 给 coding agent 的 README,直接接代码。
8. **细节体感好** —— 有区分的加载态、owl 助手、TIP 文案、模式分区清晰。

---

## 8. 可挑剔 / 局限

- **澄清问题偏多**:一次问 10+ 题,对"就想快点出图"的用户略重(它有 `Decide for me` 兜底,但默认全问)。
- **多方向是横向堆叠的长页**:方向多了要上下滚动比对,缺一个"并排对齐看细节"的对比视图。
- **强对话依赖**:所有改动走"说/圈选",对"我就想拖一下这个间距"的精细手动调整,不如传统画布直接(它有 Edit/Tweaks 模式,但重心仍在对话)。
- **生成是"出整版"**:看不到像我们幻灯片那样"逐页流式浮现"的过程(它的 Scrambling 更接近黑盒)——这恰好是**我们产品可差异化**的地方。
- **闭环偏 Anthropic 生态**:handoff 最顺的下游是 Claude Code。

---

## 9. 给我们产品的可迁移范式(逐条映射)

| Claude Design 的做法 | 搬到我们 AI 幻灯片产品的哪里 |
|---|---|
| 开场 `questions_v2` 结构化澄清 + 智能默认 | **输入起步页**:篇幅/受众/语气用 chip 选,前置收集 → 省掉对话式 brief gate |
| todo 三态 + 流式 + 验证 agent 可见 | **工作台副驾**(已做);把后端 reviewer/style-guard 的自检也暴露成可见事件 |
| 多方向 + 差异说明 + 收敛追问 | **选模板 / 选布局**岔路口:给可比方案 + 一句差异 + 一键选 |
| Mark up 点局部圈选改(批注 vs 让AI改 两按钮) | **触点④改这一页**:点元素→说改什么→局部重生;区分"记一笔"与"让AI改" |
| handoff 包(意图+规范+产物+说明) | **交付**:给用户可继续编辑/重生的结构化包,不只一份成品 |
| 有区分的加载态 + 具体文案 | **渲染阶段**:"正在排第3页图文双栏"而非 Loading;再叠加我们的逐页骨架填充 |

**一句话**:Claude Design 把"agentic 生产交互"在设计域打了个样;我们要做的,是把同一套(澄清→计划→流式→人在回路→局部改→交付)**搬到幻灯片域**,并在它偏弱的"逐页流式可见"上做出我们的高光。
